# CC-Visual Roadmap

> 从"看见会话"升级到"能处理会话"——把 AI 编程会话雷达变成真正的效率工具。

## 当前状态 (v1.0)

- Express + Canvas 像素风实时可视化
- 支持 Claude Code / Codex 双数据源
- 会话三态识别：Running / Waiting / Idle
- SSE 4秒轮询 + 工具级状态展示
- 右侧面板：当前工具 + 最近调用历史
- 侧边栏会话列表（按状态排序）
- Windows 窗口聚焦
- **已迁移至 TypeScript**：后端拆分为 `src/server/` 下 8 个模块（index / config / scanner / enrichment / tool-parser / codex / focus-window / routes），前端拆分为 `src/client/` 下多个模块 + `src/client/components/` + `src/client/utils/`，共享类型位于 `src/shared/`

## S1: 状态筛选 + 来源筛选 + 搜索框

**目标**：会话多了以后侧边栏不崩溃，能快速定位目标会话。

### 功能点

- 侧边栏顶部增加筛选按钮组：All / Running / Waiting / Idle
- 点击切换，高亮当前筛选项，显示各状态计数 badge
- 按 Claude Code / Codex 分组或筛选，支持同时选多个来源
- 侧边栏顶部搜索框，支持模糊匹配项目名、Session ID、当前工具名
- 实时过滤，无需回车

### 改动范围

`src/client/sidebar.ts`（侧边栏渲染逻辑）、`public/css/style.css`（筛选栏样式）

### 验收标准

- [ ] 点击 Running/Waiting/Idle 按钮，列表只显示对应状态的会话
- [ ] 各按钮旁的 badge 数字与实际会话数一致
- [ ] 点击 Claude Code / Codex 来源按钮可筛选，支持多选
- [ ] 搜索框输入项目名关键词，列表实时过滤匹配结果
- [ ] 搜索框输入 Session ID 片段，能匹配到对应会话
- [ ] 清空搜索框后列表恢复完整显示
- [ ] 状态筛选与搜索框可叠加使用（如：筛选 Running + 搜索 "my-project"）

## S4: 数据源健康检查

**目标**：遇到"为什么没显示"时能自行诊断，尽早提升可用性。

> 注：此迭代从原 Phase 7 提前，因为诊断能力越早具备越好。

### 功能点

- HUD 栏增加诊断入口（齿轮图标）
- 诊断面板显示：

| 检查项 | 状态 |
|--------|------|
| Claude 数据目录 | 存在 / 缺失 |
| Codex 数据目录 | 存在 / 缺失 |
| Codex SQLite 可读 | 正常 / 异常 |
| 最近扫描时间 | 2秒前 |
| SSE 连接状态 | 已连接 / 断开 |
| 已扫描 JSONL 文件数 | 15 |
| 被过滤会话数（超时） | 3 |

- 后端新增 `GET /api/health` 返回各数据源状态
- SSE 断线时显示重连提示，自动重连 + 重连计数
- 连接恢复后 toast 通知

### 改动范围

`src/server/routes.ts`（健康 API `GET /api/health`）、`src/client/panel.ts`（诊断面板）、`src/client/utils/sse-client.ts`（重连逻辑）、`public/css/style.css`（诊断面板样式）

### 验收标准

**正常路径：**

- [ ] HUD 栏有诊断入口图标，点击展开诊断面板
- [ ] 诊断面板正确显示 Claude / Codex 数据目录状态
- [ ] 诊断面板显示最近扫描时间和已扫描文件数
- [ ] `GET /api/health` 返回 JSON，包含字段：`claudeDir.exists`、`codexDir.exists`、`codexSqlite.readable`、`lastScanAt`、`scannedFiles`、`filteredSessions`、`sseClients`
- [ ] 诊断面板可关闭，不影响正常使用

**故障注入验收：**

- [ ] `config.json` 中 `claudeDir` 指向不存在的目录时，诊断面板显示"缺失"且页面不崩溃
- [ ] Codex SQLite 文件不存在或被锁定时，诊断面板显示"异常"且其他数据源正常工作
- [ ] 手动断开网络（或关闭后端）后，页面在 5 秒内显示断线重连提示
- [ ] SSE 连续重连失败 3 次以上时，显示"连接失败，请检查服务"而非无限静默重试
- [ ] 网络恢复后自动重连成功，显示 toast 提示，诊断面板 SSE 状态回到"已连接"
- [ ] `/api/health` 中任意字段异常时，诊断面板对应行标红，其余行正常显示

## S5: 会话详情 — 回复摘要 + 错误信息

**目标**：点开一个角色就能看到"它最近说了什么"和"有没有出错"。

### 功能点

- 右侧面板增加"最近回复"区域
- 截取最后一条 assistant 纯文本的前 200 字符
- 后端 `/api/transcript` 扩展返回 `lastAssistantText`
- 检测最近工具调用中 `status === 'error'` 的条目
- 以红色高亮显示错误摘要
- 后端扫描 JSONL 提取最近错误

### 改动范围

`src/server/routes.ts`（API 扩展）、`src/server/tool-parser.ts`（JSONL 错误提取）、`src/client/panel.ts`（右侧面板渲染）、`public/css/style.css`（回复 + 错误样式）

### 验收标准

- [ ] 选中会话后右侧面板显示"最近回复"区域，内容不超过 200 字符
- [ ] 超过 200 字符时末尾显示省略号 `...`
- [ ] 会话没有 assistant 回复时，"最近回复"区域显示占位提示（如"暂无回复"）
- [ ] `/api/transcript` 响应中包含 `lastAssistantText` 字段
- [ ] Claude Code 和 Codex 两种数据源的会话均能正确返回 `lastAssistantText`
- [ ] 有错误工具调用时，面板中以红色高亮显示错误摘要
- [ ] 无错误时不显示错误区域（非显示空白区域）
- [ ] 切换选中会话后，面板内容正确刷新

## S6b: 会话详情 — 阶段推断 tag

**目标**：一眼看出会话"正在做什么类型的事"。

### 功能点

- 根据最近工具类型推断阶段标签，在角色名牌下方显示阶段 tag
- 推断规则（v1，只看最近 1 个工具）：

| 最近工具 | 阶段标签 |
|---------|---------|
| Grep / Glob / Read | `搜索中` |
| Edit / Write | `编辑中` |
| Bash | `执行命令` |
| （会话状态为 Waiting） | `等你确认` |
| 其他或无工具 | `工作中` |

- `等你确认` 优先级最高：无论最近工具是什么，只要会话状态为 Waiting，标签覆盖为 `等你确认`

### 改动范围

`src/client/character.ts`（阶段 tag 渲染）、`public/css/style.css`（tag 样式）、`src/server/enrichment.ts`（可选：在 enrichSession 中计算阶段）

### 验收标准

**规则准确性验收：**

- [ ] 最近工具为 Grep 时，显示 `搜索中`
- [ ] 最近工具为 Glob 时，显示 `搜索中`
- [ ] 最近工具为 Edit 时，显示 `编辑中`
- [ ] 最近工具为 Write 时，显示 `编辑中`
- [ ] 最近工具为 Bash 时，显示 `执行命令`
- [ ] 最近工具为 Read 时，显示 `搜索中`
- [ ] 最近工具为未映射类型时，显示 `工作中`
- [ ] 会话状态为 Waiting 时，无论最近工具是什么，标签为 `等你确认`

**UI 验收：**

- [ ] 阶段 tag 显示在角色名牌下方
- [ ] 阶段 tag 随工具调用变化实时更新
- [ ] 无工具调用历史的会话显示 `工作中`

### 技术风险

阶段推断是启发式算法。v1 只看最近 1 个工具类型，规则表固定。后续可迭代为看最近 N 个工具的序列模式，但需要收集真实使用反馈后再优化。

## S7: 操作闭环

**目标**：从看板直接跳进处理动作，减少上下文切换。

### 功能点

- 右侧面板按钮：打开项目所在文件夹（后端 `POST /api/open-folder`）
- Session ID / Thread ID 旁增加复制按钮，点击复制到剪贴板 + toast 提示
- 右侧面板按钮：在新标签页查看完整 transcript
- Pin 会话到侧边栏顶部持续跟踪，Pin 状态存 localStorage
- 被 Pin 的会话即使变成 Idle 也不消失

### 改动范围

`src/server/routes.ts`（`POST /api/open-folder`）、`src/client/panel.ts`（按钮交互）、`src/client/sidebar.ts`（Pin 逻辑）、`public/css/style.css`（按钮样式）

### 口径定义

- **Transcript 查看**：Claude Code 和 Codex 双数据源均需支持。Claude Code 读 JSONL，Codex 读 SQLite conversations 表。两种来源的 transcript 渲染到同一个查看页面，格式统一。
- **打开目录**：使用会话的 `cwd` / 项目路径字段，调用系统默认文件管理器。

### 验收标准

- [ ] 点击"打开目录"按钮，系统文件管理器打开对应项目目录
- [ ] 点击复制按钮，Session ID 复制到剪贴板并显示 toast
- [ ] 点击"查看 Transcript"按钮，新标签页展示完整会话记录
- [ ] Claude Code 来源的会话 transcript 正确展示
- [ ] Codex 来源的会话 transcript 正确展示
- [ ] 点击 Pin 按钮后会话固定在侧边栏顶部
- [ ] Pin 状态刷新后保持
- [ ] 被 Pin 的会话变为 Idle 后仍然显示在顶部
- [ ] 取消 Pin 后会话回到正常排序位置

## S8a: 本地通知 MVP

**目标**：会话进入 Waiting 时主动提醒，不用一直盯着地图。

### 功能点

- 使用 Notification API，会话进入 Waiting 时推送桌面通知
- 首次使用时请求通知权限
- 内置像素风提示音（短促 8-bit beep），Waiting 触发时播放
- HUD 栏一键静音按钮，状态存 localStorage

### 改动范围

`src/client/notifications.ts`（通知 + 音效逻辑）、`public/css/style.css`（静音按钮样式）

### 验收标准

**正常路径：**

- [ ] 会话进入 Waiting 时浏览器弹出桌面通知，内容包含项目名
- [ ] 同时播放像素风提示音
- [ ] 首次使用弹出通知权限请求

**异常路径：**

- [ ] 通知权限被拒绝时不报错，降级为仅 HUD 提醒 + 音效
- [ ] 浏览器不支持 Notification API 时静默降级，不影响其他功能
- [ ] 同一会话短时间内多次切换状态时不重复通知（防抖）

**交互验收：**

- [ ] 点击静音按钮后不再播放提示音
- [ ] 静音状态刷新后保持
- [ ] 静音不影响桌面通知（静音只控制音效）

## S9a: 基础统计

**目标**：提供当日使用概览和项目级工具分布。

### 功能点

- 今日会话总数 / 活跃会话数
- 今日工具调用总数
- Claude vs Codex 使用占比（简单条形图）
- 项目级工具调用分布（柱状图）
- 按工具类型聚合计数
- 统计数据从内存聚合，不引入持久化

### 口径定义

- **"今日"**：以服务器本地时区的 00:00-23:59 为一天
- **"活跃会话"**：状态为 Running 或 Waiting 的会话
- **"工具调用总数"**：从当前活跃 + 最近 Idle 会话的 JSONL 中聚合，不跨天累积

### 改动范围

`src/client/game.ts`（HUD 统计）、`src/client/components/stats.ts`（统计面板）、`src/server/routes.ts`（统计聚合 API）、`public/css/style.css`（统计样式）

### 验收标准

- [ ] HUD 栏显示今日会话总数和活跃会话数
- [ ] HUD 栏显示今日工具调用总数
- [ ] 统计面板显示 Claude vs Codex 使用占比条形图
- [ ] 点击项目可查看该项目的工具调用分布柱状图
- [ ] 统计数据随会话活动实时更新
- [ ] 页面刷新后统计从当前活跃数据重新聚合（不持久化）

## S9c: 日统计持久化与查询

**目标**：数据跨天留存，支持历史查询。

> 前置条件：开始前做技术 spike，评估 SQLite vs JSON 文件存储的取舍。

### 功能点

- 后端按天归档会话统计
- 新增 `POST /api/stats/archive` 手动/定时归档当天数据
- 新增 `GET /api/stats/daily?date=YYYY-MM-DD` 查询历史统计
- 存储方案：SQLite（首选）或 JSON 文件（降级）

### 改动范围

`src/server/routes.ts`（统计 API）、新增 `src/server/stats-store.ts`（存储层）、`package.json`（新增 `better-sqlite3` 依赖，如选 SQLite）

### 验收标准

**API 验收：**

- [ ] `POST /api/stats/archive` 归档当天数据并返回 200
- [ ] `GET /api/stats/daily?date=2026-04-08` 返回 JSON，包含字段：`totalSessions`、`toolCalls`、`bySource`、`byProject`
- [ ] 查询不存在的日期返回 404 或空结构（不报错）
- [ ] 重复归档同一天数据时覆盖而非追加

**持久化验收：**

- [ ] 重启服务后历史数据不丢失
- [ ] 跨天数据各自独立，互不影响

**降级验收：**

- [ ] SQLite 文件损坏或不可用时，服务启动不崩溃，降级为无持久化模式
- [ ] 降级模式下 `GET /api/stats/daily` 返回明确的错误信息（如 `{"error": "storage unavailable"}`）
- [ ] 降级不影响其他 API 和实时功能

### 技术风险

引入 SQLite 是唯一的架构变更点，新增 `better-sqlite3` 依赖。spike 需评估：

- Windows/macOS/Linux 下 `better-sqlite3` 编译是否顺畅
- 数据量预期（每天几十条统计 vs 逐条工具调用）是否真需要数据库
- JSON 文件按天存储 `data/stats/2026-04-08.json` 是否已经够用

## 实施原则

1. **每个迭代独立可交付** — 完成一个迭代就是一个可用版本
2. **前端优先** — S1-S3 纯前端改动，不碰后端，见效最快
3. **渐进增强** — 每个功能都有降级方案，不破坏现有体验
4. **像素风一致** — 新增 UI 元素保持 Press Start 2P 字体 + 暗色霓虹配色
5. **最小改动** — 优先改现有文件，不无必要不新建文件
6. **先验后做** — 每个迭代先写验收 checklist，再动手实现
7. **风险前置** — 有技术风险的功能先做 spike 验证
8. **双源一致** — 涉及数据展示的功能须同时验证 Claude Code 和 Codex 两种数据源
9. **验故障不只验成功** — 涉及外部依赖（文件系统、网络、权限）的迭代，验收必须包含异常/降级路径
